<project_specification>
  <project_name>raws - AWS CLI Reimplementation in Rust</project_name>

  <overview>
    Build a complete reimplementation of the AWS CLI v2 in Rust, including a reimplementation of
    botocore (the low-level AWS SDK). The project is called "raws" and produces a binary also called
    "raws" that is a full drop-in replacement for the "aws" CLI.

    The goal is FULL PARITY with the AWS CLI. This means supporting ALL ~418 AWS services and
    replicating the behavior of the "aws" command exactly. The approach to achieving this is:

    1. Build the core infrastructure (signing, credentials, 4 protocol implementations, endpoint
       resolution, config loading) — this is the hard part and is done by hand.
    2. Load the REAL botocore JSON service model files at runtime, exactly like the Python CLI does.
       The models describe every operation, its parameters, and its response shapes. The CLI
       interprets these models dynamically using serde_json::Value — no code generation needed.
       Since all AWS services use one of 4 protocols, a correct model loader + protocol layer
       automatically supports all 418 services with zero per-service code.
    3. Copy the actual botocore service model files (the same JSON files used by the Python AWS
       CLI) into the models/ directory. These are ~100MB of JSON covering every AWS service.
    4. Build the CLI driver that dynamically discovers all services from models/ and exposes them
       as subcommands, exactly like "aws <service> <operation>".
    5. Implement the same customizations the Python CLI has (S3 high-level commands, configure,
       cloudformation deploy, etc.) — these are hand-written on top of the core infrastructure.

    The real botocore model files live at:
      $AWS_CLI_SRC/botocore/data/
    (AWS_CLI_SRC is defined in .env — load it before running any commands)

    This directory contains ~418 service subdirectories, each with:
      - service-2.json (API model: operations, shapes, metadata)
      - paginators-1.json (pagination config)
      - waiters-2.json (polling waiters, not all services have this)
      - endpoint-rule-set-1.json (endpoint resolution rules)
      - completions-1.json (shell completions data)

    It also contains global files:
      - endpoints.json (~40K lines, maps services+regions to endpoint URLs)
      - partitions.json (AWS partition definitions: aws, aws-cn, aws-us-gov)

    The binary must support every command the real AWS CLI supports:
      raws <any-service> <any-operation> [--params...]
      raws s3 cp/mv/sync/ls/rm/mb/rb          (high-level S3 commands)
      raws configure                           (interactive credential setup)
      raws --output json|table|text|yaml
      raws --region <region>
      raws --profile <profile>
      raws --query <jmespath-expression>
      raws --debug
  </overview>

  <runtime_model_interpretation>
    This project uses runtime model interpretation, NOT code generation. This is the same
    approach used by the Python botocore library. Here's how it works:

    1. At startup, the CLI scans the models/ directory to discover available services.
    2. When a user invokes "raws sts get-caller-identity":
       a. Load models/sts/2011-06-15/service-2.json into a serde_json::Value
       b. Parse it into a ServiceModel struct (operations, shapes, metadata)
       c. Look up the "GetCallerIdentity" operation
       d. Parse CLI arguments based on the operation's input shape definition
       e. Build a serde_json::Value (or HashMap) from the parsed arguments
       f. Pass the value to the protocol serializer which uses the shape definitions
          to produce the correct HTTP request (form-encoded, JSON, XML, etc.)
       g. Sign, send, receive response
       h. Pass the response to the protocol parser which uses the output shape
          definitions to produce a serde_json::Value
       i. Format and print the result

    The key data types:
    - ServiceModel: parsed from service-2.json, contains metadata, operations, shapes
    - Shape: enum of structure/string/integer/list/map/etc. with type-specific metadata
    - Operation: input shape ref, output shape ref, HTTP method/URI, errors
    - serde_json::Value: the universal intermediate representation for request/response data

    Why this works:
    - The CLI boundary is always strings (user types args, output is text)
    - All type validation is done against the model at runtime
    - All serialization/deserialization is driven by shape definitions
    - No per-service Rust code is needed — the same generic code handles every service
    - This is exactly how Python botocore works, and it's proven at scale
  </runtime_model_interpretation>

  <service_model_format>
    Service models are JSON files that describe AWS service APIs. The model loader
    must parse these at runtime to drive request serialization and response parsing.

    <service_2_json>
      The main model file. Structure:

      {
        "version": "2.0",
        "metadata": {
          "apiVersion": "2011-06-15",
          "endpointPrefix": "sts",
          "globalEndpoint": "sts.amazonaws.com",
          "protocol": "query",
          "serviceAbbreviation": "AWS STS",
          "serviceFullName": "AWS Security Token Service",
          "serviceId": "STS",
          "signatureVersion": "v4",
          "uid": "sts-2011-06-15"
        },
        "operations": {
          "GetCallerIdentity": {
            "name": "GetCallerIdentity",
            "http": { "method": "POST", "requestUri": "/" },
            "input": { "shape": "GetCallerIdentityRequest" },
            "output": {
              "shape": "GetCallerIdentityResponse",
              "resultWrapper": "GetCallerIdentityResult"
            }
          }
        },
        "shapes": {
          "GetCallerIdentityRequest": {
            "type": "structure",
            "members": {}
          },
          "GetCallerIdentityResponse": {
            "type": "structure",
            "members": {
              "UserId": { "shape": "userIdType" },
              "Account": { "shape": "accountType" },
              "Arn": { "shape": "arnType" }
            }
          },
          "userIdType": { "type": "string" },
          "accountType": { "type": "string" },
          "arnType": { "type": "string" }
        }
      }

      Shape types to handle:
      - "structure" -> serde_json::Value::Object (fields from "members")
      - "string" -> Value::String (may have "enum" list for validation)
      - "integer" / "long" -> Value::Number
      - "float" / "double" -> Value::Number
      - "boolean" -> Value::Bool
      - "timestamp" -> Value::String (formatted per protocol)
      - "blob" -> base64 encoded Value::String
      - "list" -> Value::Array (has "member" with shape ref)
      - "map" -> Value::Object (has "key" and "value" with shape refs)

      Member modifiers:
      - "required": [...] on a structure shape lists required members
      - "location": "header" / "querystring" / "uri" -> where to put the value
      - "locationName": "x-amz-foo" -> the actual header/query/xml name
      - "streaming": true -> the member is a streaming body (for S3 GetObject etc.)

      Protocol types (from metadata.protocol):
      - "query" -> Query/form-encoded POST body (STS, IAM, EC2, AutoScaling)
      - "json" -> AWS JSON protocol with X-Amz-Target header (DynamoDB, SWF)
      - "rest-json" -> REST with JSON body (Lambda, API Gateway)
      - "rest-xml" -> REST with XML body (S3, Route53, CloudFront)
    </service_2_json>

    <paginators_1_json>
      Defines how to paginate list operations:

      {
        "pagination": {
          "ListBuckets": {
            "result_key": "Buckets"
          },
          "ListObjectsV2": {
            "input_token": "ContinuationToken",
            "output_token": "NextContinuationToken",
            "limit_key": "MaxKeys",
            "result_key": "Contents"
          }
        }
      }
    </paginators_1_json>

    <waiters_2_json>
      Defines polling waiters:

      {
        "version": 2,
        "waiters": {
          "BucketExists": {
            "delay": 5,
            "maxAttempts": 20,
            "operation": "HeadBucket",
            "acceptors": [
              { "expected": 200, "matcher": "status", "state": "success" },
              { "expected": 404, "matcher": "status", "state": "retry" }
            ]
          }
        }
      }
    </waiters_2_json>
  </service_model_format>

  <protocols_detail>
    Each protocol defines how to serialize a request and deserialize a response.
    The serializers work with serde_json::Value and the model's shape definitions
    to produce the correct wire format.

    <query_protocol>
      Used by: STS, IAM, EC2, AutoScaling, ELB, CloudWatch, SNS, SQS

      Request serialization:
      - HTTP method: POST
      - Content-Type: application/x-www-form-urlencoded
      - Body is form-encoded: Action=GetCallerIdentity&Version=2011-06-15
      - Nested structures use dot notation: Filter.1.Name=instance-id&Filter.1.Value.1=i-123
      - Lists use positional notation: InstanceId.1=i-123&InstanceId.2=i-456

      Response parsing:
      - Response is XML
      - Wrapped in <{OperationName}Response><{OperationName}Result>...</>...</>
      - Parse XML, map element names to shape member names
      - Error responses: <ErrorResponse><Error><Code>...</Code><Message>...</Message></Error></ErrorResponse>

      EC2 variant:
      - Same as query but error format differs slightly
      - Uses <Response><Errors><Error>... format
    </query_protocol>

    <json_protocol>
      Used by: DynamoDB, KMS, CloudTrail, CodeCommit

      Request serialization:
      - HTTP method: POST
      - Content-Type: application/x-amz-json-1.0 (or 1.1, from metadata.jsonVersion)
      - Header: X-Amz-Target: DynamoDB_20120810.ListTables
        (format: metadata.targetPrefix + "." + operation.name)
      - Body: JSON serialization of input value

      Response parsing:
      - Body is JSON, deserialize directly to serde_json::Value
      - Error responses: {"__type": "ResourceNotFoundException", "message": "..."}
    </json_protocol>

    <rest_json_protocol>
      Used by: Lambda, API Gateway, Kinesis

      Request serialization:
      - HTTP method from operation.http.method
      - URI template from operation.http.requestUri (e.g., /2015-03-31/functions/{FunctionName})
      - Members with location=uri go into the URL path
      - Members with location=querystring go into query params
      - Members with location=header go into HTTP headers
      - Remaining members go into JSON body
      - If operation has no body members, no body is sent

      Response parsing:
      - Members with location=header come from response headers
      - Members with location=statusCode come from HTTP status
      - Remaining members parsed from JSON body
      - Streaming members: response body is the raw stream
    </rest_json_protocol>

    <rest_xml_protocol>
      Used by: S3, Route53, CloudFront

      Request serialization:
      - HTTP method from operation.http.method
      - URI template from operation.http.requestUri
      - Members with location=uri/querystring/header same as rest-json
      - Remaining non-streaming members serialized as XML body
      - XML uses locationName for element names (or member name if not specified)
      - S3 uses virtual-hosted style: {bucket}.s3.{region}.amazonaws.com

      Response parsing:
      - Headers and status code same as rest-json
      - Body parsed as XML
      - Error responses: <Error><Code>NoSuchBucket</Code><Message>...</Message></Error>

      S3 specifics:
      - Uses "s3" signature version (variant of SigV4 with different payload handling)
      - Unsigned payload for PutObject: x-amz-content-sha256: UNSIGNED-PAYLOAD
      - Virtual hosted bucket addressing by default
    </rest_xml_protocol>
  </protocols_detail>

  <sigv4_signing>
    All AWS API calls must be signed with Signature Version 4.

    Algorithm:
    1. Create a canonical request:
       HTTPMethod\n
       CanonicalURI\n
       CanonicalQueryString\n
       CanonicalHeaders\n
       SignedHeaders\n
       HashedPayload

       - CanonicalURI: URI-encode each path segment
       - CanonicalQueryString: sort query params by name, URI-encode keys and values
       - CanonicalHeaders: lowercase header names, trim values, sort by name, one per line
         ending with \n. Must include "host" and "x-amz-date". Include "x-amz-security-token"
         if using session credentials.
       - SignedHeaders: semicolon-delimited list of header names from CanonicalHeaders
       - HashedPayload: SHA256 hex digest of the request body (or "UNSIGNED-PAYLOAD" for S3)

    2. Create string to sign:
       AWS4-HMAC-SHA256\n
       Timestamp\n              (ISO 8601 basic: 20230101T000000Z)
       Scope\n                  (20230101/us-east-1/sts/aws4_request)
       Hash(CanonicalRequest)   (SHA256 hex)

    3. Calculate signing key:
       kDate    = HMAC-SHA256("AWS4" + SecretKey, Date)      (Date = "20230101")
       kRegion  = HMAC-SHA256(kDate, Region)
       kService = HMAC-SHA256(kRegion, Service)
       kSigning = HMAC-SHA256(kService, "aws4_request")

    4. Calculate signature:
       signature = HexEncode(HMAC-SHA256(kSigning, StringToSign))

    5. Add Authorization header:
       Authorization: AWS4-HMAC-SHA256
         Credential={AccessKey}/{Scope},
         SignedHeaders={SignedHeaders},
         Signature={signature}

    Also add these headers before signing:
    - Host: service.region.amazonaws.com
    - X-Amz-Date: 20230101T000000Z
    - X-Amz-Security-Token: {session_token} (only if present)
    - X-Amz-Content-Sha256: {payload_hash} (required for S3)

    The signing key can be cached per date+region+service since it only depends on those.

    TEST VECTORS: Use the AWS SigV4 test suite from the AWS documentation to verify your
    signing implementation. At minimum, verify that signing an STS GetCallerIdentity request
    with known credentials produces the expected Authorization header.
  </sigv4_signing>

  <credentials>
    The credential provider chain tries these sources in order:

    1. Environment variables:
       - AWS_ACCESS_KEY_ID
       - AWS_SECRET_ACCESS_KEY
       - AWS_SESSION_TOKEN (optional)

    2. Shared credentials file (~/.aws/credentials):
       [default]
       aws_access_key_id = AKIA...
       aws_secret_access_key = secret...

       [my-profile]
       aws_access_key_id = AKIA...
       aws_secret_access_key = secret...

       The profile is selected by --profile CLI arg or AWS_PROFILE env var.

    3. Config file (~/.aws/config):
       [default]
       region = us-east-1
       output = json

       [profile my-profile]
       region = eu-west-1
       role_arn = arn:aws:iam::123456789012:role/MyRole
       source_profile = default

       Note: profile sections in config use "profile " prefix except for [default].

    4. EC2 Instance Metadata Service (IMDS) - can be stubbed initially.

    Credentials struct:
      - access_key_id: String
      - secret_access_key: String
      - session_token: Option<String>

    The CredentialProvider trait:
      async fn resolve(&self) -> Result<Credentials, CredentialError>
  </credentials>

  <endpoint_resolution>
    The real AWS CLI uses two mechanisms for endpoint resolution:

    1. endpoints.json — A ~40K-line JSON file that maps (partition, service, region) to
       endpoint URLs. This file is in models/endpoints.json (copied from botocore/data/).
       It handles all the special cases: global services, dual-stack, FIPS, S3 virtual-hosted, etc.

    2. endpoint-rule-set-1.json — Per-service rule engine (newer mechanism). Each service
       directory may contain this file with rules for resolving endpoints.

    For the initial implementation, use a simple fallback strategy:
    - Default: https://{endpointPrefix}.{region}.amazonaws.com
      (endpointPrefix comes from the service model's metadata)
    - Parse endpoints.json for the accurate mapping — it covers special cases like:
      * S3: https://{bucket}.s3.{region}.amazonaws.com (virtual-hosted)
        or https://s3.{region}.amazonaws.com/{bucket} (path-style)
      * STS global: https://sts.amazonaws.com
      * IAM global: https://iam.amazonaws.com
      * Partition-specific: *.amazonaws.com.cn for China, etc.
    - Full endpoint-rule-set-1.json engine can come in later milestones.

    The endpoints.json format:
    {
      "partitions": [
        {
          "partition": "aws",
          "defaults": {
            "hostname": "{service}.{region}.{dnsSuffix}",
            "protocols": ["https"],
            ...
          },
          "dnsSuffix": "amazonaws.com",
          "regionRegex": "^(us|eu|ap|sa|ca|me|af|il)\\-\\w+\\-\\d+$",
          "regions": { "us-east-1": { "description": "US East (N. Virginia)" }, ... },
          "services": {
            "s3": {
              "defaults": { "protocols": ["http", "https"], "signatureVersions": ["s3"] },
              "endpoints": { "us-east-1": { "hostname": "s3.us-east-1.amazonaws.com" }, ... }
            },
            ...
          }
        }
      ]
    }
  </endpoint_resolution>

  <cli_driver>
    The CLI binary works as follows:

    1. Parse global arguments: --region, --profile, --output (json|table|text), --debug
    2. First positional arg is the service name (e.g., "sts", "s3api", "s3")
    3. Second positional arg is the operation name (e.g., "get-caller-identity")
    4. Remaining args are operation parameters (e.g., --bucket my-bucket)

    For auto-generated service commands (ALL ~418 services):
    - At startup, discover all service model directories in models/
    - Each service directory becomes a subcommand (e.g., "raws sts", "raws ec2", "raws lambda")
    - The service name is derived from the directory name (kebab-case as-is)
    - Load the service model on demand when that service is invoked
    - Find the operation matching the kebab-case command name
    - Parse operation arguments dynamically from the model's input shape
      (each member becomes --kebab-case-name, required members are required args)
    - Serialize using the service's protocol (determined by metadata.protocol)
    - Sign the request
    - Send via HTTP client
    - Deserialize the response
    - Format output (json/table/text)

    This dynamic approach means ALL services work automatically once the core
    infrastructure (protocols, signing, model loading) is correct — no per-service code needed.

    For custom commands (s3 cp, s3 ls, s3 sync, configure, cloudformation deploy, etc.):
    - These are hand-written commands that use the core infrastructure internally
    - They override or extend the auto-generated service commands
    - "raws s3" is a custom namespace (the auto-generated S3 API commands live under "raws s3api")
    - "raws s3 cp" uses the transfer logic in cli/customizations/s3/transfer.rs
    - "raws configure" interactively prompts for credentials and writes config files
    - See awscli/customizations/ in the Python CLI for the full list of ~60 customizations

    Operation name mapping:
    - CLI uses kebab-case: get-caller-identity, list-buckets, describe-instances
    - Model uses PascalCase: GetCallerIdentity, ListBuckets, DescribeInstances
    - Convert by splitting on uppercase boundaries and joining with hyphens

    Parameter name mapping:
    - CLI uses --kebab-case: --bucket-name, --instance-ids
    - Model uses PascalCase for member names: BucketName, InstanceIds
    - Same conversion as operation names

    Output formatting:
    - JSON (default): pretty-printed JSON with 4-space indent
    - Table: ASCII table with headers from shape member names
    - Text: tab-separated values, one row per item
  </cli_driver>

  <output_formatting>
    <json_format>
      Pretty-print the response using serde_json with 4-space indentation.
      This is the default and simplest format.

      Example (raws sts get-caller-identity --output json):
      {
          "UserId": "AIDAEXAMPLE",
          "Account": "123456789012",
          "Arn": "arn:aws:iam::123456789012:user/alice"
      }
    </json_format>

    <text_format>
      Tab-separated values. Top-level keys become columns.
      Lists are expanded one item per line.

      Example (raws sts get-caller-identity --output text):
      AIDAEXAMPLE    123456789012    arn:aws:iam::123456789012:user/alice
    </text_format>

    <table_format>
      ASCII table with box-drawing characters.

      Example (raws sts get-caller-identity --output table):
      -------------------------------------------------------
      |              GetCallerIdentity                       |
      +----------+--------------+---------------------------+
      | UserId   | Account      | Arn                       |
      +----------+--------------+---------------------------+
      | AIDAEX.. | 123456789012 | arn:aws:iam::123456...    |
      +----------+--------------+---------------------------+
    </table_format>
  </output_formatting>

  <s3_high_level_commands>
    The "raws s3" subcommand provides high-level file operations:

    raws s3 ls [s3://bucket[/prefix]]
    - No args: list all buckets
    - With bucket: list objects in bucket with optional prefix
    - --recursive: list all objects recursively

    raws s3 cp <source> <dest> [--recursive]
    - Local to S3: raws s3 cp file.txt s3://bucket/key
    - S3 to local: raws s3 cp s3://bucket/key file.txt
    - S3 to S3: raws s3 cp s3://src-bucket/key s3://dst-bucket/key
    - Uses multipart upload for files > 8MB (configurable threshold)

    raws s3 mv <source> <dest>
    - Same as cp but deletes the source after successful copy

    raws s3 rm s3://bucket/key [--recursive]
    - Delete object(s)

    raws s3 sync <source> <dest>
    - Sync directory to/from S3
    - Only copies changed files (by size and last modified time)
    - --delete: remove files in dest that don't exist in source

    raws s3 mb s3://bucket-name
    - Create bucket

    raws s3 rb s3://bucket-name [--force]
    - Remove bucket (--force empties it first)

    These commands use the transfer logic in cli/customizations/s3/ which manages:
    - Multipart uploads (split large files into parts, upload concurrently)
    - Multipart downloads (range GETs for large files)
    - Progress reporting
    - Retry logic per part
  </s3_high_level_commands>
